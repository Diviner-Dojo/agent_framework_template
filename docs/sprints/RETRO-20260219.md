## What Needs Improvement

1. **Feedback loop completed for the first time in this session**: This is the first `/retro` ever run. All 20 adopted patterns remained PENDING until this evaluation. The Adoption Audit Feedback Loop pattern was itself adopted but never exercised. While the gap is now closed (ADR-0011 adds specialist dispatch to both `/retro` and `/meta-review`), the delay highlights that learning loops need scheduling, not just tooling.

2. **Adoption log accuracy — CORRECTED**: Initial verification reported anti-patterns in only 1/9 agents, but this was a **case-sensitive grep bug** (`Anti-pattern` vs actual `Anti-Patterns`). Re-verification confirmed all 9/9 agents have anti-patterns sections with 5+ prohibitions each. Similarly, activation triggers are present in all 9/9 agents — 7 use "Activate for..." and 2 use equivalent "Use when/this..." phrasing. The adoption log claims are accurate. **Lesson: Verification scripts must use case-insensitive matching.**

3. **Education gates not yet warranted** *(revised per specialist feedback)*: 0 walkthroughs and 0 quizzes is a finding, but Principle #6 says gates are "proportional to complexity and risk." This sprint was a framework-design sprint where the developer authored every architectural decision. Education gates will be more meaningful during application development sprints (Phase 1 code builds) where the developer is learning new patterns.

4. **Orphaned discussion — RESOLVED**: DISC-20260219-035210-analyze-claude-agentic-framework had `status=open` with 0 events. Root cause investigation confirmed: the claude-agentic-framework analysis ran successfully (10 patterns evaluated, 6 adopted into adoption log), but `write_event.py` was never called for specialist turns. This occurred at 03:52 UTC — before ADR-0009 mandated capture pipeline integration for `/analyze-project`. The analysis results survive in the adoption log (indirect evidence) but the specialist reasoning is lost. Discussion closed with explanatory note. ADR-0009 prevents recurrence.

5. **Collaboration mode calibration for next sprint** *(reframed per specialist feedback)*: 85% structured-dialogue usage (11/13 discussions) is not necessarily overuse for this sprint — framework infrastructure decisions genuinely warranted multi-round exchange. However, the next sprint (application code) will have different risk profiles. For low-risk analyses and known-solution discussions, ensemble mode should be the default. The risk-tiering guidance in `review_gates.md` already maps low-risk to ensemble — the gap is that this guidance only applies to `/review`, not to `/deliberate` and `/analyze-project`.

6. **Stale references accumulated silently**: `commit_protocol.md` still referenced `src/` and `tests/` (Python-era paths) and ruff/pytest tools. Fixed in this session, but highlights that documentation drift goes undetected without explicit auditing.

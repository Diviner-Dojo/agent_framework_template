## Required Changes (Blocking)

### RESOLVED: Unguarded `_primary.stop()` in `FallbackTtsService.stop()`

**Identified by**: architecture-consultant (Round 1), confirmed REVISE Round 2
**File**: `lib/services/fallback_tts_service.dart:66`
**Root cause**: `stop()` called `_primary.stop()` unconditionally. If primary was in a broken
state (the reason fallback was activated), a thrown exception from `_primary.stop()` would
prevent `_fallback.stop()` from executing, leaving fallback audio playing — an audio resource
leak that defeats the decorator's reliability contract.
**Fix applied**: Wrapped `_primary.stop()` in `try { } on Exception { }` so fallback stop
always executes regardless of primary state.
**Regression test added**: `'stop() calls fallback even when primary throws (regression)'`
in `test/services/fallback_tts_service_test.dart`.

## Strengths

- **FallbackTtsService decorator pattern is well-chosen.** Implements `TextToSpeechService`,
  transparent to all consumers. The orchestrator and provider layer are unaware of fallback
  logic — clean separation of concerns matching ADR-0015/22.
- **Eager fallback initialization** — both services initialized before any speak() call.
  System TTS is lightweight and always ready. Avoids cold-start failure on first utterance
  after primary failure.
- **Regression ledger is exemplary.** All 5 fixes logged with root cause, file, fix
  description, and test location in `memory/bugs/regression-ledger.md`.
- **Fix 2 guard placement is architecturally correct.** Moving `journalOnlyMode` to after
  `_routeByIntent()` correctly separates "structured intent handling" from "AI conversational
  follow-up" — the right semantic boundary.
- **Fix 4 `_completeOnboardingIfNeeded()` ordering** is correctly sequenced and documented
  inline with a comment explaining why state must not be cleared first.
- **Fix 3 async patterns are correct.** `unawaited()` for PopScope sync constraint,
  `await stop()` in async discard path, `StateError` guard in `dispose()`.
- **daQuery false alarm resolved in Round 2.** `_handleDayQuery` is local-data-only;
  the concern about journal-only mode AI leakage applies only to `_handleRecallQuery`.

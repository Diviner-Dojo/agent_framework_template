## Blocking Findings (Both Resolved In-Review)

### B1 — BootReceiver permission does not protect against third-party explicit intents [RESOLVED]

**Severity**: Blocking (Security Misconfiguration)
**Specialist**: security-specialist
**Location**: android/app/src/main/AndroidManifest.xml:65-68

`android:permission="android.permission.RECEIVE_BOOT_COMPLETED"` uses a **normal-level permission**. Android grants normal-level permissions automatically at install time to any app that declares them — no user consent required. A malicious app holding this permission could send explicit intents to BootReceiver. The manifest comment claiming this "prevents third-party apps from sending explicit intents" was factually incorrect, creating a false security assumption for future developers.

**Fix applied**: Changed `android:exported="true"` → `"false"` and removed the `android:permission` attribute. Per Android documentation, system broadcasts (BOOT_COMPLETED, MY_PACKAGE_REPLACED) are delivered by the OS regardless of exported state on API 26+. Third-party explicit intents are now fully blocked. Updated manifest comment.

### B2 — updateNotificationId has zero test coverage [RESOLVED]

**Severity**: Blocking (Missing Test)
**Specialist**: qa-specialist
**Location**: lib/database/daos/task_dao.dart (updateNotificationId), test/database/task_dao_test.dart

`TaskDao.updateNotificationId()` is new production code that closes the boot-restore loop — if it silently does nothing (wrong `where` clause), stale notification IDs persist in the database indefinitely and every subsequent reboot will retry the failed reschedule. Zero tests existed.

**Fix applied**: Added 3 tests to `test/database/task_dao_test.dart` in a new `updateNotificationId` group:
1. Happy path: insert task with `notificationId=1042`, call `updateNotificationId('t-upd-notif', 2000)`, verify `notificationId==2000` and `rowsUpdated==1`
2. Null path: set `notificationId` to null, verify column is null
3. No-row path: call with non-existent taskId, verify `rowsUpdated==0`

Quality gate: 7/7 at 80.1% (89 total task_dao tests).

## Advisory Findings (8 Open)

### A1 — deleteTasksBySession cancellation wiring not tested
**Severity**: Medium | **Specialist**: qa-specialist

`TaskDao.deleteTasksBySession()` iterates tasks and calls `cancelNotification()` for each — this path is not covered by the new `_FakeScheduler` tests. A refactor accidentally removing the loop would not be caught.

**Recommendation**: Add test asserting `cancelledIds` contains all session task notification IDs after `deleteTasksBySession()`.

### A2 — rescheduleFromTasks !_initialized early-return not tested
**Severity**: Medium | **Specialist**: qa-specialist

The `!_initialized || tasks.isEmpty` early-return path is directly observable without platform channels. No test exists for this method at all.

**Recommendation**: Add one test: construct uninitialized service, call `rescheduleFromTasks([])`, assert empty list returned.

---

### A3 — Arithmetic-only wrap tests do not exercise production code
**Severity**: Medium | **Specialist**: qa-specialist
**Location**: test/services/notification_scheduler_service_test.dart:127-180

The wrap-around test (lines 127-150) and boundary-reset test (lines 152-180) operate entirely on local `const` variables. Production code in `_nextNotificationId()` is never invoked. An off-by-one bug in the service would not be caught. Test names imply behavioral coverage they do not provide.

**Recommendation**: Add clarifying comments: "arithmetic contract only — production wrap path requires on-device integration test (initialize() requires real platform channels)."

---

### A4 — PENDING_CREATE status inclusion not tested
**Severity**: Low | **Specialist**: qa-specialist
**Location**: test/database/task_dao_test.dart (getTasksWithPendingReminders group)

`getTasksWithPendingReminders` uses `isNotIn([completed])`, including PENDING_CREATE, but no test verifies PENDING_CREATE tasks are returned. A future change to `isIn([active])` would be a regression not caught.

**Recommendation**: Add one test with `status: TaskStatus.pendingCreate` verifying the task appears in results.

---

### A5 — Counter persistence test comment should acknowledge ordering assumption
**Severity**: Low | **Specialist**: qa-specialist
**Location**: test/services/notification_scheduler_service_test.dart:101-125

The broad `catch (_)` works correctly only if `_nextNotificationId()` completes before the plugin throws. This ordering assumption is not validated.

**Recommendation**: Strengthen comment: "Valid only because _nextNotificationId() is awaited and persists before _plugin.zonedSchedule() is called. If call order is inverted, this test must be updated."

---

### A6 — PlatformException not caught in rescheduleFromTasks (deferred A-4)
**Severity**: Low | **Specialist**: security-specialist
**Location**: lib/services/notification_scheduler_service.dart

When `SCHEDULE_EXACT_ALARM` is revoked (Android 12+), `zonedSchedule()` throws `PlatformException`. This is uncaught — `notificationBootRestoreProvider` enters `AsyncError` silently, stale `notificationId` remains in DB, and the silent failure loop repeats every cold start. Minor marginal exposure: task titles may appear in crash logs via the unhandled exception.

**Recommendation**: Add `on PlatformException { /* log */ }` catch in `rescheduleFromTasks()` and clear the stale `notificationId` from the task row. Document in ADR-0033. (Was advisory A-4 in REV-20260304-074715 — still open.)

---

### A7 — Bidirectional coupling should be documented in ADR-0033
**Severity**: Low | **Specialist**: architecture-consultant
**Location**: lib/database/daos/task_dao.dart:22, lib/services/notification_scheduler_service.dart:30

`task_dao.dart` imports the service (for constructor parameter), and `notification_scheduler_service.dart` imports database models (`Task`). The bidirectional coupling is pragmatic and testability is maintained via `_FakeScheduler`, but it should be documented as intentional.

**Recommendation**: Add a note to ADR-0033 section on service boundaries documenting this coupling was deliberate.

### A8 — taskDaoProvider asymmetry comment missing
**Severity**: Low | **Specialist**: architecture-consultant
**Location**: lib/providers/database_provider.dart

`taskDaoProvider` is the only DAO provider importing from `notification_providers.dart`. No comment explains why.

**Recommendation**: Add brief comment: "Requires scheduler for notification cancellation on delete/complete — differs from other DAO providers which depend only on databaseProvider."

## Strengths

- **ADR-0033 fidelity**: On-launch approach faithfully implemented. BootReceiver correctly a no-op; Dart layer handles all rescheduling. ADHD-safe: silently skips past-due tasks, `requestPermissionIfNeeded: false` prevents unsolicited prompts.
- **_FakeScheduler pattern**: Correct subclass approach — only overrides `cancelNotification`. DAO operations run against a real in-memory database. Tests are not over-mocked.
- **Test isolation**: `setUp`/`tearDown` with fresh `NativeDatabase.memory()` per test. `_FakeScheduler` re-instantiated per group. No shared mutable state.
- **`ref.read` in FutureProvider**: Correct for once-per-cold-start semantics (consistent with `llmAutoLoadProvider` pattern). Non-autodispose correctly caches the result for the app lifetime.
- **Idempotent boot-restore**: `scheduleNotification` overwrites existing alarms. Partial execution is safe — next cold start re-runs from scratch.
- **Specific exception catches**: `PastReminderTimeException`, `NotificationPermissionDeniedException` caught individually. No bare catch. TOCTOU race condition comment shows awareness.
- **Privacy preserved**: Task titles excluded from all log statements. `NotificationVisibility.private` at notification level. `allowBackup=false` on app.

## Required Changes (blocking)

**B1 — RESOLVED during review: Missing false-positive test for new word-count expansion territory.**

The word-count wildcard `(\s+[\w-]+){0,4}` is broader than the prior `.{0,15}` char-count wildcard. The SPEC-20260303-010332 risk table explicitly required "explicit false-positive tests for new coverage." Architecture-consultant (Round 2) identified that the ADR-0020 §8 confirmation gate is a downstream mitigation that does not prevent incorrect routing to the extraction LLM — and therefore does not substitute for test coverage.

Fixed: test added documenting the accepted behavior of `"Add context to the call summary"` → `calendarEvent` (3 intervening words before `call`, a `_calendarEventNouns` member). Test comment explains the tradeoff: the confirmation gate prevents auto-creation; the user makes the final decision. Total test count in the new regression group: 10 (was 9).

## Recommended Improvements (advisory)

**A1 (Architecture/QA, Medium): `_hasStrongCalendarSignal` constructs `RegExp` per call**
Pre-existing pattern, all other strong-signal helpers have the same issue. Now materially larger with the `[\w-]+` and `{0,4}` quantifier. Recommendation: extract to `static final _strongCalendarSignalPattern` field alongside other named patterns. No behavior change. Deferred from build checkpoint.

**A2 (QA, Low): `@Tags(['regression'])` absent from both regression groups**
`flutter_test` `group()` does not support a `tags` parameter. Library-level `@Tags` would tag all tests in the file. Project pattern uses library-level `@Tags` only in standalone regression files. The gap is consistent with the existing PR #56/57 regression group. Recommendation: create a dedicated `intent_classifier_regression_test.dart` file with library-level `@Tags` and move regression groups there in a future sprint.

**A3 (Architecture/Docs, Low): ADR-0030 missing template fields**
`risk_level`, `confidence`, `tags` frontmatter fields absent. `## References` section should be `## Linked Discussion` per template. Notably ADR-0029 also omits these fields — applying this as blocking would be inconsistent with prior ADRs. Fixed in-review: Alternative D label changed from "chosen" to "selected approach IF Step 3 initiates" for consistency with `status: proposed`. Recommendation: add `risk_level: low`, `confidence: 0.70`, `tags: [developer-input, capture-stack]` and a `## Linked Discussion` section in a future sprint.

**A4 (Docs, Medium): Advisory A6 from REV-20260302-232244 not addressed — carried forward**
A6 states: documentation_policy.md `## Request Context` rule has no automated enforcement — recommend adding parenthetical noting this. This advisory was out of scope per SPEC-20260303-010332 which only listed A1–A5. Advisory is now at 2 sprint carry-forward without resolution.

**A5 (Architecture, Low): `_calendarEventNouns` doc comment should name specific consumers**
Comment says "Both locations must be updated together" but doesn't name `_calendarIntentPattern` and `_hasStrongCalendarSignal` explicitly. A third call site added without referencing the constant would silently re-introduce divergence. Recommendation: amend doc comment to name both consumers.

**A6 (QA, Low): Voice preamble test confidence assertion can be tightened**
Test 5 (`"Okay add a meeting tomorrow"`) asserts `greaterThanOrEqualTo(0.5)` but actual score is `0.75` (base + temporal boost from "tomorrow"). Tighten to `greaterThanOrEqualTo(0.75)` to make temporal boost observable.

**A7 (QA, Low): "Set the call to silent" carry-forward from REV-20260302-230547 A1**
Pre-existing accepted ambiguity: this message classifies as `calendarEvent` (confirmation gate backstops). Still no pinned test documenting the behavior. Carried forward from REV-20260302-230547.

**A8 (Docs, Low): retro.md Step 5.1 write_event.py call had unquoted args**
Fixed in-review: `facilitator` and `evidence` now quoted to match other 4 files. No behavioral change (bash treats bare alphanumeric args identically), but prevents a future copy-paste trap.

## Strengths

- The `_calendarEventNouns` shared constant is the correct structural fix for the divergent-noun-list class of regression identified in PR #57. Noun-list synchronization is now enforced by the compiler rather than by convention — the type of enforcement that would have prevented PR #56 and PR #57 from occurring.
- Both `_calendarIntentPattern` and `_hasStrongCalendarSignal` were updated in the same commit, satisfying the SPEC requirement and directly addressing the root cause of PR #57.
- The asymmetric anchor design (`\b` in `_calendarIntentPattern` for voice preamble support; `^` in `_hasStrongCalendarSignal` for short-message guard precision) is architecturally correct and clearly documented in both source comments and test file header.
- Test coverage structure is disciplined: regression group clearly separates brand-name path (exercises `_calendarIntentPattern` directly), short-message guard path (exercises `_hasStrongCalendarSignal`), voice preamble path, word-count boundary, and false-positive guards. Post-review: 10 tests including the new FP territory documentation test.
- All 5 advisory items A1–A5 from REV-20260302-232244 are demonstrably closed with specific file locations. A6 is correctly carried forward (out of spec scope). Advisory tracking is clean and traceable.
- ADR-0030 is substantively more than a stub: it includes a concrete evaluation gate with observable signals (Signal A: specialist echo, Signal B: framing drift, Signal C: disposition activity), four alternatives with rejection reasoning, and explicit conditional framing. The "if not implemented, this ADR remains proposed" language correctly serves the ADR immutability principle.
- The INVARIANT comment text is identical across all five command files, and the cold-start fallback guidance introduces a queryable tag (`context-brief-cold-start`) that makes uninstrumented invocations observable in the pipeline.

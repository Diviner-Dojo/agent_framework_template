## Request Context

- **What was requested**: Fix intent classifier to handle 'Add a Google Calendar meeting' (user-reported on device: phrase bypassed intent routing, Claude responded 'I can't access your Google Calendar')
- **Files/scope**: `lib/services/intent_classifier.dart`, `test/services/intent_classifier_test.dart`, `memory/bugs/regression-ledger.md`
- **Developer-stated motivation**: The phrase "a Google Calendar " (19 chars) exceeded the `.{0,15}` character limit in `_calendarIntentPattern`, causing silent fallthrough to Claude despite Google Calendar being connected in Settings
- **Explicit constraints**: No false positives (e.g., 'add a note about my team meeting' must not trigger calendar intent); conservative, targeted fix

## Recommended Improvements (advisory)

**A1 (QA):** Add false-positive guard test — `'Add a note about my team meeting' → journal`. The developer-stated constraint is correctly enforced by the regex (`^add` anchor + required `calendar` keyword) but is not codified in a test. A future pattern widening could silently break it. One-liner addition.

**A2 (QA):** Add test for `'Add it to Google Calendar'` (no possessive "my"/"the") to confirm the optionality chain in Change 1 works end-to-end. The `(my\s+|the\s+)?` is optional in the pattern but no test exercises the non-possessive path.

**A3 (QA):** Protocol gap — `@Tags(['regression'])` is absent from the 4 new tests. The project-wide pattern uses group/test naming conventions instead of annotations. Not blocking merge but worth aligning in a future cleanup pass.

**A4 (Independent):** Other named-calendar brands (Outlook Calendar, iCloud Calendar) have the same structural failure as the original bug. "Add an Outlook Calendar meeting" — "an Outlook Calendar " = 20 chars, exceeds the new sub-pattern's `.{0,15}` left limit and still falls through to Claude. Enterprise users (Wells Fargo context) may encounter this. Recommend Sprint N+1 structural fix: replace `^add\b.{0,15}` with a word-count wildcard `^add\b(\s+\w+){0,5}` that is brand-agnostic.

**A5 (Independent):** `^add\b` anchor breaks for voice preambles. "Okay, add a Google Calendar meeting" fails the `^` anchor. Voice mode is live and in active use. Pre-existing limitation, not introduced by this fix. Recommend: consider removing `^` anchor from the `add` sub-patterns or switching to `\b` word boundary (matching the approach used in the "want to / need to" alternative). Worth a TODO comment at minimum.

**A6 (Independent):** No defense-in-depth at the routing boundary. If a message containing "calendar" falls through to `IntentType.journal`, no log or analytics event signals the gap. The original bug was detected only via user report. A routing-boundary check that emits a debug warning when high-signal keywords (calendar, meeting, schedule, appointment) appear in messages routed to journal/Claude would surface future pattern gaps in QA rather than production.

## Structural Note on Record

The `independent-perspective` correctly identifies that `.{0,N}` character counting is structurally brittle for intent patterns with variable-length modifiers. This fix correctly addresses the reported symptom. The class of problem — any named-calendar modifier exceeding the character limit — remains open. Sprint N+1 consideration: replace `^add\b.{0,15}\b(...)` with word-count wildcard `^add\b(\s+\w+){0,5}\s+\b(meeting|appointment|event|...)`.

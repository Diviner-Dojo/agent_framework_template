## Required Changes (Blocking)

### B1 — Reverse-scored items not re-reversed before correlation normalization ✅ RESOLVED IN-REVIEW

**Found by**: independent-perspective
**Files**: `lib/providers/questionnaire_providers.dart`, `lib/providers/check_in_trend_provider.dart`

**Root cause**: `CheckInHistoryEntry` had no per-item `isReversed` map. `checkInTrendProvider` normalized raw answers directly without applying `scaleMax + scaleMin - raw` for reversed items (e.g., Anxiety, `isReversed: true`). High-anxiety days (raw=10) normalized to 1.0 — treated as "maximum good" by Pearson correlation — producing semantically inverted correlation directions for any dimension paired with Anxiety.

**Clinical impact**: A user with consistently high anxiety + low mood would see "Your Anxiety and Mood scores tend to move together" — a false positive correlation. The correct clinical picture is opposite: high anxiety tracks with low mood wellness.

**Resolution applied**:
1. Added `Map<int, bool> itemIsReversed` to `CheckInHistoryEntry` with doc comment explaining its role in the trend pipeline
2. Populated `itemIsReversedCache` from `getAllItemsForTemplate` items (`.isReversed` field) in `checkInHistoryProvider` alongside the existing `itemTextCache`
3. Applied reversal in `checkInTrendProvider` before calling `normalizeAnswer()`:
   ```dart
   final isReversed = entry.itemIsReversed[answer.itemId] ?? false;
   final rawValue = isReversed
       ? (entry.scaleMax + entry.scaleMin - answer.value!).toDouble()
       : answer.value!.toDouble();
   ```
4. Regression test added: `'reverse-scored items are re-reversed before normalization (regression)'`
5. Regression ledger entry added to `memory/bugs/regression-ledger.md`

## Recommended Improvements (Advisory)

### A1 — Y-axis on rolling averages chart is unexplained (UX/Medium)
Add subtitle below "Rolling averages" section header: `'Values normalized to 0–1 (1 = highest recorded)'`. Users see 0.0/0.5/1.0 with no explanation of what these numbers mean.

### A2 — hasSufficientData gate (2 days) admits users before chart is useful (UX/Medium)
The 7-day rolling window shows mostly-null content for users with 2–6 check-in days. Consider graying out window toggles below their data threshold, or defaulting to "All" when < 7 days recorded.

### A3 — Raw r-value in correlation tile subtitle is statistical jargon (UX/Medium)
`'strong, move together — r = 0.78 (14 days)'` exposes the Pearson coefficient. The strength label and direction already communicate magnitude. Remove `r = x.xx`, keep `pairedCount` in days.

### A4 — Correlation empty state gives no actionable threshold (UX/Medium)
`'Not enough shared check-in days to compute correlations yet.'` → Change to `'Correlations appear after 5 or more days with data for the same dimensions.'`

### A5 — History chart SegmentedButton has shrinkWrap tap target (UX/Medium)
Line ~344: `tapTargetSize: MaterialTapTargetSize.shrinkWrap` removes 48dp minimum enforcement. Remove `shrinkWrap` from `SegmentedButton.styleFrom`.

### A6 — Two `_shortLabel` implementations with divergent logic (Architecture/Medium)
`CorrelationService._shortLabel`: stopword-filter, lowercase, 12-char cap. Screen `_shortLabel`: keyword-mapping, capitalized, 10-char cap. Same question text produces different output in narrative insights vs chart legends. Consolidate to one function.

### A7 — `_normalizeValue` vs `normalizeAnswer` degenerate case asymmetry (QA/Medium)
Screen-level `_normalizeValue` returns 0.0 for `scaleMin == scaleMax`. `CorrelationService.normalizeAnswer` returns 0.5. Difference is undocumented and untested. Either add a comment explaining the intentional difference or align the behavior.

### A8 — `hasMissingDataWarning` per-correlation path untested (QA/Medium)
A correlation with `r ≥ 0.5` but `pairedCount` 5–9 (above correlation threshold, below 10) sets `hasMissingDataWarning: true` on a normal narrative insight card. This boundary has no test. Add: `pairedCount: 7`, `r: 0.75`, `totalDays: 14` — assert flag is `true` and narrative text is still present.

### A9 — Rolling averages computed in `build()` on every `setState` (Performance/Low)
`_buildRollingSection()` calls `svc.rollingAverages()` for each item on every rebuild. Acceptable at current expected volumes (<180 days). Flag for revisit if frame drops observed at 365+ days. Consider pre-computing in a derived provider keyed on `(data, window)`.

### A10 — |r|≥0.4 threshold may be too strict for ADHD high-variability users (Independent/Low)
ADHD is characterized by high intra-individual variability. "No strong patterns detected yet" may appear for most users most of the time, making the feature feel empty. Rolling averages already provide useful content from day 2; correlation insights are secondary depth.

## Strengths

- **CorrelationService is exemplary for testability**: pure Dart, `const` constructor, zero Flutter deps, all boundary conditions handled with null-return guards.
- **Missing-data as `null` throughout entire pipeline**: never imputed as 0. Rigorously maintained from `DailyItemValues` through `rollingAverages` to `generateInsights`. Validated end-to-end by provider integration tests.
- **ADHD clinical UX compliance is thorough**: "possible relationship" framing, no gap language, no streak counters, neutral color palette (avoids red/green evaluative connotation), 3-insight cap, amber tint for low-data warnings (not red alarm).

- **Sort-assert with `debug-mode` conditional test**: `skip: dart.vm.product` correctly reflects that assertions are release-mode-invisible. Well-structured debug-only precondition enforcement.

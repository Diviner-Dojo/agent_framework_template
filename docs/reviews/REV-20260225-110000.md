## Summary

Phase 11: Google Calendar + Reminders integration reviewed across 15+ new and modified files. Five specialists identified 4 blocking findings and 18+ advisory findings. The most critical issue is a missing timezone field in Google Calendar API calls that would cause events to display at wrong times for all non-UTC users. The overall architecture is sound — ADR-0020 compliance is thorough, the injectable callable pattern is well-applied, and the dual state machine design is correct. However, four issues must be addressed before merge.

## Findings by Specialist

- **BLOCKING**: CalendarServiceException at `google_calendar_service.dart:190` interpolates `e.message` from Google API response — leaks server error detail to callers. Use fixed template with status code only.
- **BLOCKING**: `confirmCalendarEvent()` TOCTOU race — state cleared optimistically before DB insert allows double-tap to create duplicate events. Guard with `isWaitingForAgent` check.
- Advisory: `updateStatus`/`updateSyncStatus` accept raw String — add validation or enum type.
- Advisory: `rawUserMessage` synced verbatim to Supabase — privacy risk for sensitive data.
- Advisory: `googleCalendarServiceProvider` always returns null — fragile trust boundary.
- Advisory: `signIn()` swallows all exceptions as null — errors indistinguishable from cancellation.
- Advisory: Voice confirmation 8s ambient audio window.
- Advisory: `_cancelPendingEvents` fetches all sessions globally.
- Advisory: `duration_minutes` Infinity edge case in extraction.

## Required Changes Before Merge

1. **Add timezone field to EventDateTime** (independent-perspective): In `google_calendar_service.dart`, pass the user's timezone string to `gcal.EventDateTime(dateTime: ..., timeZone: ...)` in both `_createEventImpl` and `_createReminderImpl`. Without this, ALL non-UTC users get events at wrong times. Consider passing timezone from the session's `timezone` field or from `DateTime.now().timeZoneName`.

2. **Sanitize CalendarServiceException messages** (security-specialist): In `google_calendar_service.dart:190` and `:234`, replace `'Google Calendar API error ${e.status}: ${e.message}'` with `'Google Calendar API error (HTTP ${e.status})'`. Log `e.message` at debug level only.

3. **Enforce 5-event pending cap** (independent-perspective): In `session_providers.dart` `_handleCalendarIntent()` and `_handleReminderIntent()`, check `calendarEventDao.countPendingForSession(sessionId)` before proceeding. If >= 5, return early with an appropriate message per ADR-0020 §7.

4. **Guard confirmCalendarEvent against double invocation** (security-specialist): Add `if (state.isWaitingForAgent) return;` at the top of `confirmCalendarEvent()` to prevent TOCTOU race from double-tap or concurrent voice + UI confirmation.

## Recommended Improvements (Non-Blocking)

1. **Add `cancelEventsForSession` DAO method** — single SQL update instead of fetch-all + filter + per-event update (architecture + performance).
2. **Batch calendar event UPSERTs in `syncPendingCalendarEvents`** — match the existing message UPSERT pattern (performance).
3. **Add composite index on `(status, sync_status)`** — optimize `getEventsToSync` query (performance).
4. **Add `countAllPendingEvents()` aggregate** to DAO — avoid materializing rows for count (performance).
5. **Extract `_clearPendingIntents()` helper** — ensure `dismissCalendarEvent` and `dismissReminder` clear all pending fields atomically (architecture).
6. **Add tests for SessionNotifier calendar flow** — `confirmCalendarEvent`, `dismiss`, `defer`, `_cancelPendingEvents` (QA).
7. **Add tests for voice orchestrator calendar confirmation** — yes/no/timeout paths, `_isAffirmative`, `_formatTimeForSpeech` (QA).
8. **Consider excluding `rawUserMessage` from Supabase sync** — privacy improvement; data exists in messages table already (security + independent).
9. **Fix EventExtractionService header comment** — replace "ConversationLayer pattern" with accurate description (architecture).
10. **Apply range validation in regex extraction layer** — match LLM layer's validation contract (architecture).
11. **Add past-tense guard to `_calendarIntentPattern`** — prevent journaling about scheduling from triggering extraction (independent).
12. **Address optimistic success message** — delay "Added to calendar" until after API completion, or distinguish "saved locally" vs "synced to Google" (independent).

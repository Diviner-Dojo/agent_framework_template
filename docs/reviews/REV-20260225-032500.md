## Summary

Phase 9 adds photo integration to the journal app: camera/gallery capture, EXIF stripping via background isolate, inline photo messages, a photo gallery screen, cloud sync to Supabase Storage, and cascade delete support. The implementation follows ADR-0018 closely with strong security fundamentals (EXIF strip, UUID path validation, transactional cascade delete). Seven blocking findings were identified across all five specialists, with the most critical being: orphaned photos on session deletion, publicly accessible cloud URLs for private photos, and missing accessibility semantics. Seventeen advisory findings cover sync wiring gaps, test coverage holes, UX polish, and minor architecture inconsistencies.

## Findings by Specialist

### QA Specialist
- **B1**: `deleteSessionCascade` in `session_providers.dart` does not pass `PhotoDao` — orphaned photo records and files on session delete
- **A4**: UUID regex accepts non-UUID strings (e.g., `---`, `a-`)
- **A5**: session_detail_screen_test photo-message mapping assertion structurally weak
- **A6**: `deleteSessionPhotos` and `calculateTotalSize` have no unit tests
- **A7**: `Isolate.run` can throw (OOM, disk full) but `processAndSave` only handles null return
- Confidence: 0.88

### Architecture Consultant
- **B1**: Session delete and `discardSession` do not clean up photos (corroborates QA)
- **A1**: Photo sync not wired into session end flow — `uploadSessionPhotos`/`syncPendingPhotos` never called
- **A2**: `localPath` stores absolute path but ADR and schema docs say relative
- **A3**: `SessionDetailScreen` directly instantiates DAOs instead of using providers
- **A13**: Dynamic type casts in `journal_session_screen.dart` photo handling (fallback `List<dynamic>`)
- Confidence: 0.88

### Security Specialist
- **B2**: `getPublicUrl` on a private Supabase bucket generates permanently public URLs for private journal photos — must use `createSignedUrl` or store canonical path only
- **B3**: `deletePhotoFile` accepts arbitrary `localPath` with no path confinement validation — potential sandbox file deletion via corrupted DB record
- **A16**: Local DB and photos unencrypted at rest (existing gap, higher risk with photos)
- **A17**: No `UPDATE` RLS policy on `storage.objects` for `journal-photos` bucket
- Confidence: 0.92

### Independent Perspective
- **B2**: Public URL exposure corroborates security finding
- **B4**: Silent failure when `processAndSave` returns null — no user error feedback (corroborates UX)
- **B7**: Hero tag collision when same photo visible in both session screen and gallery simultaneously
- **A15**: Cascade delete file-then-DB ordering enforced by comment only, not code structure
- Pre-mortem scenarios: public photo exposure, silent photo loss on HEIC/low-memory, OOM crash, double-sync race

### UX Evaluator

## Required Changes Before Merge

1. **B1 — Fix cascade delete to include photos** (QA, Architecture): Update `deleteSessionCascade` helper in `session_providers.dart` and `SessionNotifier.discardSession` to pass `PhotoDao` and call `PhotoService.deleteSessionPhotos(sessionId)` before the DB cascade. Add regression test.

2. **B2 — Replace `getPublicUrl` with signed URLs or canonical path storage** (Security, Independent): In `sync_repository.dart:242`, replace `getPublicUrl()` with `createSignedUrl()` using short TTL, or store only the canonical storage path (`userId/photos/photoId.jpg`) in `cloudUrl` and generate signed URLs on demand at display time.

3. **B3 — Add path confinement to `deletePhotoFile`** (Security): Validate that the resolved `localPath` starts with the expected `photos/` directory under `getApplicationSupportDirectory()` before deleting.

4. **B4 — Show error feedback when photo processing fails** (UX, Independent): Add a SnackBar or error message when `processAndSave` returns null in `_capturePhoto`.

5. **B5 — Wire `isProcessing` state into preview dialog** (UX): Show the processing spinner during `processAndSave` execution. Make the dialog stay open during processing, or show a brief loading indicator before the photo appears in chat.

6. **B6 — Add `Semantics` to photo interactive elements** (UX): Wrap photo `GestureDetector` in `Semantics` widget in both `chat_bubble.dart` and `photo_gallery_screen.dart` with descriptive labels.

7. **B7 — Prevent Hero tag collision** (Independent): Use unique Hero tags that include the screen context (e.g., `'photo-gallery-$photoPath'` vs `'photo-chat-$photoPath'`) to avoid duplicate Hero tags when both screens are in the widget tree.

3. **A3**: Use provider-based DAO access in `SessionDetailScreen` instead of direct construction

5. **A5**: Strengthen session_detail_screen_test photo-message assertions

## Strengths

- EXIF stripping via decode-resize-reencode is architecturally sound and correctly implemented
- Path traversal prevention at write time via UUID regex validation is thorough (7 test cases)
- Constructor injection pattern (ADR-0007) consistently applied to `PhotoDao`
- Schema migration v3 is correct with proper index creation for both fresh and upgrade paths
- Test isolation exemplary — in-memory databases, fake-based testing, `addTearDown` for cleanup
- `SyncResult` data class provides honest API with separate success/failure/error tracking
- Cascade delete correctly wrapped in drift transaction
- Hero animation tags consistent across all three rendering surfaces
- Broken image fallback implemented uniformly across all image display points
- `context.mounted` checks correctly placed after all async navigation calls
- Delete confirmation dialog follows Material 3 destructive action pattern

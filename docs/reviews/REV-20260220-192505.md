## Summary

Retroactive review of Phase 4 (Cloud Sync & Optional Auth) — PR #9 with 35 files, 3222 insertions, 285 tests, 80.4% coverage. The optional auth model is correctly implemented: the app degrades gracefully when Supabase is not configured, RLS is enabled on all tables, secrets are properly managed, and the offline-first guarantee is never compromised. However, 4 blocking findings require attention: the UPSERT code path has zero test execution (mock returns null client), RLS policies lack explicit WITH CHECK clauses, the Edge Function's PROXY_ACCESS_KEY fallback increases attack surface now that JWT auth is live, and the correlated subquery in the journal_messages RLS policy will degrade at scale. Additionally, 15 advisory findings across all specialists identify improvements to test coverage, error handling, performance, and code organization.

## Required Changes Before Merge

Since PR #9 is already merged, these are required changes to address before the next commit/PR:

1. **Add explicit `WITH CHECK` clauses to RLS policies** (security-specialist F2): All three table policies use `FOR ALL` with only `USING`. Add explicit `WITH CHECK` clauses that mirror the `USING` expressions. This prevents silent regression if policies are ever modified.

2. **Fix UPSERT test coverage** (qa-specialist F1-F3): The authenticated test group's mock returns `null` client, so the actual upload path is never executed. Either:
   - Extract `_uploadSession` into an injectable interface, or
   - Create a spy that records whether upload was called, or
   - Add partial-failure and exception-path tests with a mock that actually exercises the UPSERT guard

3. **Evaluate PROXY_ACCESS_KEY deprecation** (security-specialist F1): Now that JWT auth is live, the fallback path is a permanent second attack surface. Either:
   - Remove the fallback entirely (breaking change for unauthenticated mode), or
   - Add minimum entropy check (`proxyAccessKey.length < 32` → reject), or
   - Document the rationale for keeping both paths in ADR-0012

4. **Add `user_id` column to `journal_messages`** (performance-analyst F1): The correlated subquery in the RLS policy re-evaluates per row. For a user with 1000+ messages, this becomes measurable. Add a `user_id` column and rewrite the policy to a direct equality check. Apply same fix to `entry_embeddings`.

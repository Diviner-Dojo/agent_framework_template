## Analysis

The root cause was a three-layer short-circuit: the DAO, repository, and provider all returned early when `query` was empty, regardless of active filters. The UI also showed the pre-search state under the same condition. The fix is correctly applied at all four sites with the guard `query.trim().isEmpty && !filters.hasActiveFilters`. The DAO fix appropriately uses `Constant(true)` rather than an early return, because the query builder requires an expression.

The message search skip for filter-only browse is correct (no keyword to match in message content) and well-optimized (two async round-trips avoided). The snippet skip (`Future.value(<String>[])`) is synchronous and allocates nothing.

The `Constant(true)` fallback path (when all constraints are null) is a clean, readable sentinel. The comment in `searchSessions` accurately explains the filter-only design intent.

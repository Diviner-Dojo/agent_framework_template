## Summary

Phase 4B (Android Quick Capture home screen widget) and Phase 4C (passive weather metadata capture) are well-implemented at the service layer, correctly following the injectable-callable pattern, fire-and-forget contract, and ADR-0019 privacy chain. However, 5 blocking findings were identified: an Intent injection vulnerability in the widget dispatch path, a missing connectTimeout that enables resource accumulation under hostile networks, an undocumented weather sync boundary decision, missing ADR-0034 for all Phase 4B/4C architectural decisions, and a functional gap where widget taps are silently dropped when the app is already backgrounded (the most common real-world usage pattern).

## Request Context

- **What was requested**: Review Phase 4B (Android Quick Capture home screen widget) and Phase 4C (passive weather metadata capture at session start) implementation
- **Files/scope**: lib/services/weather_service.dart, lib/services/widget_launch_service.dart, lib/providers/weather_providers.dart, lib/providers/last_capture_mode_provider.dart, lib/providers/session_providers.dart, lib/ui/screens/session_list_screen.dart, lib/app.dart, lib/database/app_database.dart, lib/database/tables.dart, lib/database/daos/session_dao.dart, test/ui/session_list_screen_test.dart, android/app/src/main/kotlin/com/divinerdojo/agentic_journal/QuickCaptureWidget.kt, android/app/src/main/AndroidManifest.xml
- **Developer-stated motivation**: ADHD effortless capture — one-tap widget opens last-used mode; passive weather context enriches session metadata
- **Explicit constraints**: Fire-and-forget weather must never block session start; SharedPreferences cross-layer coupling is intentional design

### Security Specialist

- A1: Missing connectTimeout — only receiveTimeout set; TCP connection can hang for OS default (~75s) under captive portal
- A2-A4: Certificate pinning, exported receiver, unencrypted weather data — all accepted given null-fallback contract and non-PII payload

### Performance Analyst
- **REVISE** — Confidence: 0.91
- B1 (HIGH): Missing connectTimeout on Dio — receiveTimeout fires only after connection is established; under stalled TCP, connection phase can hang for OS default (~75s), accumulating event-loop slots across sessions
- A1: Dio instantiation per call — negligible at 1-5 calls/day
- A2: No deduplication window for rapid session starts — accepted trade-off
- Strengths: Fire-and-forget correctly defers to event loop, schema migration is O(1), single UPDATE for all weather columns is optimal

### Independent Perspective
- **REVISE** — Confidence: 0.82
- B1 (HIGH): Widget launch dropped on warm-start — _checkWidgetLaunch() in app.dart runs only in initState() with _widgetLaunchChecked guard. App backgrounded → user taps widget → onNewIntent() fires in Kotlin → Flutter never re-reads channel → widget tap silently dropped. This is the MOST COMMON usage scenario.
- B2 (Medium): PendingIntent baked at widget-refresh time (not tap time) — mode can be stale for 30+ minutes; contradicts ADHD 'effortless capture' contract
- A1: No validation of mode string from widget (stale mode from old app version falls through gracefully)
- A2: Weather sync excluded without documentation — same gap as architecture-consultant
- A3: weatherDescription denormalization — stored string could diverge from future improved descriptions
- A4: No debounce guard for rapid widget taps (assistant intent path has one)

## Required Changes Before Merge

1. **B1 — Intent injection allowlist** (security-specialist, independent-perspective): Add allowlist validation in `session_list_screen.dart` ref.listen before calling `_dispatchCaptureMode`:
   ```dart
   const _kValidWidgetModes = {'text', 'voice', '__quick_mood_tap__', 'pulse_check_in'};
   if (!_kValidWidgetModes.contains(next)) return;
   ```

2. **B2 — connectTimeout on Dio** (performance-analyst, security-specialist): Add `connectTimeout: const Duration(seconds: 5)` to `Options()` in `WeatherService._defaultFetch`:
   ```dart
   options: Options(
     connectTimeout: const Duration(seconds: 5),
     receiveTimeout: const Duration(seconds: 5),
   ),
   ```

3. **B3 — Weather sync boundary decision** (architecture-consultant, independent-perspective): Resolve the ambiguity in `sync_repository.dart` — either (a) add `weatherTempC`, `weatherCode`, `weatherDescription` to `buildSessionUpsertMap` plus Supabase migration, or (b) add a unit test asserting their exclusion and a comment citing the decision. Without this, future developers cannot distinguish this from a bug.

4. **B4 — ADR-0034** (architecture-consultant): Write ADR-0034 covering: (1) Open-Meteo as weather provider (free, no API key, GDPR-compliant), (2) fire-and-forget pattern rationale and never-blocks contract, (3) sync boundary decision (explicit), (4) SharedPreferences cross-layer coupling risk acceptance and graceful degradation strategy.

5. **B5 — Warm-start widget dispatch** (independent-perspective): Add `_checkWidgetLaunch()` call in `didChangeAppLifecycleState` when `state == AppLifecycleState.resumed`, using a separate resume-path guard (resets on each resume) distinct from `_widgetLaunchChecked` (cold-start one-time guard). This mirrors the `_checkAssistantRelaunch()` pattern already in app.dart.

## Recommended Improvements (Non-Blocking)

A1. **describeWeatherCode missing test buckets** (QA): Add tests for codes 56/57 (Freezing drizzle), 66/67 (Freezing rain), 77 (Snow grains), 85/86 (Snow showers). Test comment currently claims 'all bucket boundaries' — incorrect.

A2. **Voice mode widget launch test** (QA): Add 4th Phase 4B test: `'voice mode via widget launch: sets voiceModeEnabledProvider=true and clears provider'`. Same pump structure as text-mode test.

A3. **PendingIntent staleness** (independent-perspective): Call `AppWidgetManager.updateAppWidget()` from Flutter side via MethodChannel after `lastCaptureModeProvider.setMode()` completes, so widget refreshes immediately on mode change rather than waiting for Android's periodic refresh cycle.

A4. **Stale-session guard test** (QA): Add `'does not write weather when session is discarded before API resolves'` using a Completer-based fake fetch. This is the only uncovered behavioral branch in `_captureWeatherAsync`.

A5. **WidgetLaunchService test file** (QA): Create `test/services/widget_launch_service_test.dart` with 2 tests: happy path + PlatformException returns null.

A6. **Weather test timing** (QA): Raise `Future.delayed(100ms)` in weather capture tests to `300ms` to match Phase 4B pattern and reduce CI flakiness risk.

A7. **pendingWidgetLaunchModeProvider placement** (architecture-consultant): Advisory for future — extract to dedicated file if widget channel grows to handle multiple signal types.

A8. **weatherDescription denormalization comment** (independent-perspective): Add a comment noting that `weatherDescription` is derived from `weatherCode` and that the stored string may diverge from future improved descriptions; `weatherCode` is the canonical value.

A9. **Rapid widget tap debounce** (independent-perspective): Add a debounce guard to the widget launch path analogous to `shouldProcessAssistIntent()` to prevent duplicate sessions from back-to-back taps.

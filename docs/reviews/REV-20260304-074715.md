## Summary

Combined review of two completed builds: Phase 5A Visual Identity & Theme Personalization (7 curated palettes, `ThemeState`, `ThemeNotifier`, `ChatBubbleColors` ThemeExtension, settings UI) and SPEC-20260304-061650 Scheduled Local Notifications (`NotificationSchedulerService`, schema v11 with `reminderTime`/`notificationId`/`isQuickReminder` columns, `TaskDao` notification cancellation wiring). Two blocking findings were identified and resolved in-review: `initialize()` was never called in `main.dart` (the entire notification feature would have failed at runtime), and an async exception test used non-awaited `expect()` providing zero regression protection. 13 advisories remain, with the BootReceiver reboot-survival gap (A-1) as highest priority.

## Findings by Specialist

### Architecture Consultant
- **[B-1 RESOLVED] initialize() never called in main.dart** — Without initialize(): timezone data unloaded, Android notification channel not created, ID counter not restored. Fixed: `main.dart` now constructs plugin + service, calls `initialize()` in try/catch, overrides both providers in ProviderScope following ConnectivityService pattern.
- [A-1] BootReceiver empty — ADR-0033 reboot-survival requirement unimplemented (see advisory section)
- [A-11] TaskAutoSuggestNotifier still uses legacy StateNotifier API — last remaining pre-Riverpod-2.x notifier
- [A-12] No `_initialized` guard in `scheduleNotification()` — assert as defense-in-depth
- Strengths: ADR-0033 fidelity excellent, ADR-0029 correctly evaluated, ThemeNotifier composite state is correct design, ChatBubbleColors ThemeExtension with lerp() is correct Material 3 approach
- Confidence: 0.91

### Security Specialist
- [A-4] SCHEDULE_EXACT_ALARM permission revocation (Android 12+) throws uncaught `PlatformException` from `zonedSchedule()` — task row stores `notificationId` but no OS alarm exists
- [A-5] BootReceiver `android:exported=true` without `android:permission` attribute — future `onReceive()` logic exploitable by explicit intent
- [A-13] `notificationId` + `rawUserMessage` correlation in SQLite — not remotely exploitable, `allowBackup=false` is primary mitigation
- Strengths: `NotificationVisibility.private` applied at per-notification level (authoritative), title/body never reach logcat in any mode, background isolate handler intentionally empty with clear comment, drift type-safe queries throughout
- Confidence: 0.91

### QA Specialist
- **[B-2 RESOLVED] False-positive async exception test** — `expect()` used instead of `await expectLater()` on async closure returning Future. Always passed regardless of whether exception was thrown. Fixed: changed to `await expectLater()`.
- [A-2] TaskDao `deleteTask`/`completeTask` not tested with injected mock scheduler — cancellation side-effect unasserted
- [A-3] ID wrap-around boundary (1999→1000) untested — testable via `SharedPreferences.setMockInitialValues`
- [A-8] `@Tags(['regression'])` missing on `migration_v7_test.dart`
- [A-9] `setShowDecorations()` mutator path has no dedicated test
- [A-10] `cancelAll()` has no test
- Strengths: WCAG AA contrast test across all 7 palettes × 2 modes, ThemeNotifier tests comprehensive, migration tests cover backward compat explicitly, `ChatBubbleColors.lerp()` boundary tests present
- Confidence: 0.88

### Independent Perspective
- [A-1] BootReceiver empty — ADR-0033 reboot-survival requirement not implemented; `onReceive()` is empty; comment misidentifies `onDidReceiveBackgroundNotificationResponse` callback purpose (fires on taps, not boot). Every user who migrates devices silently loses all pending task reminders.
- [A-6] `isQuickReminder` column ships with no UI routing — quick-reminder tasks appear in main task list, contradicting schema comment
- [A-7] `ReminderService` (Phase 4D) and `NotificationSchedulerService` are two separate reminder systems with different dismissal semantics, no shared state — inconsistent UX when both active
- Alternative considered: Hash-based notification IDs from `taskId.hashCode % 1000 + 1000` — eliminates SharedPreferences round-trip and counter-wrap, makes IDs recoverable after reboot. Deferred for ADR discussion.
- Confidence: 0.82

## Recommended Improvements (Non-Blocking)

**High Priority**
1. **A-1 — Implement BootReceiver reboot rescheduling** (ADR-0033 requirement): On `BOOT_COMPLETED` or app launch, query tasks where `reminderTime > now AND notificationId IS NOT NULL`, cancel old IDs and reschedule. Required before production merge.

**Medium Priority**
2. **A-4 — Handle SCHEDULE_EXACT_ALARM revocation**: Wrap `_plugin.zonedSchedule()` in `try/catch(PlatformException)`, throw typed exception, clear `notificationId` from task row.
3. **A-2 — Test TaskDao notification cancellation wiring**: Add group with mock `NotificationSchedulerService` asserting `cancelNotification()` is called with `task.notificationId` on `deleteTask`/`completeTask`.
4. **A-3 — Test ID wrap-around**: Seed `SharedPreferences` with `notification_scheduler_next_id: 1999`, verify wrap to 1000.
5. **A-5 — BootReceiver permission attribute**: Add `android:permission="android.permission.RECEIVE_BOOT_COMPLETED"` to AndroidManifest.xml receiver declaration.
6. **A-6 — Flag isQuickReminder as deferred**: Add comment to `tables.dart` and tracking note in BUILD_STATUS.md that the Reminders section UI is deferred.

**Low Priority**
7. **A-7 — Document ReminderService/NotificationSchedulerService relationship**: Add cross-reference comment between Phase 4D and Phase 5A systems; consider shared dismissal tracking in future sprint.
8. **A-8** — Add `@Tags(['regression'])` to `migration_v7_test.dart`
9. **A-9** — Add dedicated `setShowDecorations()` test
10. **A-10** — Add `cancelAll()` test
11. **A-11** — Migrate `TaskAutoSuggestNotifier` to modern `Notifier` API
12. **A-12** — Add `assert(_initialized, 'Call initialize() before scheduling')` in `scheduleNotification()`
13. **A-13** — On backup/restore, clear `notificationId` on all tasks (OS alarm state unknown after restore)

# Review: Journal-Only Voice Mode — Three Bug Fixes

## Verdict: APPROVE-WITH-CHANGES

No blocking findings. Seven advisory items. Regression tests correctly exercise all three failure conditions. Architecture is sound and ADR-0016 compliance is confirmed.

## Summary of Changes

Three bugs reported on device after v0.17.0+8 in journal-only voice mode:

1. **Bugs 1 & 2 (orchestrator stuck in processing)**: Fixed by adding `acknowledgeNoResponse()` to `VoiceSessionOrchestrator` and calling it from `SessionNotifier._resumeOrchestratorIfVoiceMode()` at the journal-only and handled-intent early exits.
2. **Bug 3 ("goodbye" ignored)**: Fixed by moving `shouldEndSession()` above the `journalOnlyMode` guard and intent routing in `sendMessage()`.

## Blocking Findings

*None.*

| A1 | Medium | missing-test | `voice_session_orchestrator.dart:498` | `acknowledgeNoResponse()` no-op contract unverified. The phase guard (non-processing → silent return) is correct but untested. A test asserting idle state is unchanged after `acknowledgeNoResponse()` would pin the contract against future refactors. |
| A2 | Medium | undocumented-assumption | `session_providers.dart:510` | `shouldEndSession()` reorder is correct but why it is safe is undocumented. An inline comment would protect future maintainers from questioning the ordering. |
| A3 | Medium | pattern-inconsistency | `voice_session_orchestrator.dart:443` | `onAssistantMessage()` guards on `idle` phase only, not `listening`. If both `acknowledgeNoResponse()` (direct) and `onAssistantMessage()` (DB stream) somehow fire concurrently, the latter would attempt to speak while already listening. Impossible in practice for journal-only paths (no ASSISTANT message inserted), but future-proofing guard would add `or phase==listening` check. |
| A4 | Low | pattern-deviation | `voice_session_orchestrator.dart:498-503` | `acknowledgeNoResponse()` does not check `isContinuousMode` before calling `_startListening()`. Push-to-talk transitions to `idle` before the `onSendMessage` callback fires, so this method will never be reached in push-to-talk mode. But the missing guard deviates from the pattern in `_commitUserTurn()` and `onAssistantMessage()`. |
| A5 | Low | missing-test | `session_providers.dart:511` | `followUpCount > 3` auto-end path in journal-only mode is untested. The reorder fix covers it, but a test would verify the `>3` branch reaches `endSession()`. |
| A6 | Low | coupling | `voice_session_orchestrator.dart` + `agent_repository.dart` | Dual-layer end-session detection: `VoiceCommandClassifier` and `AgentRepository._doneSignals` both include "goodbye". Cross-reference comments would prevent the lists from diverging silently in future. |
| A7 | Low | coverage | `session_providers.dart:1920` | `StateError` catch branch in `_resumeOrchestratorIfVoiceMode()` is dead code from a test coverage perspective. Accept as-is — `StateError` is the correct base class for Riverpod provider disposal. |

## Recommended Immediate Actions (Pre-Commit)

**A1** and **A4** are the most concrete improvements with no behavioral risk:

- **A1**: Add one test to `test/services/voice_session_orchestrator_test.dart` verifying `acknowledgeNoResponse()` is a no-op when the orchestrator is in `idle` state.
- **A4**: Add `isContinuousMode` guard to `acknowledgeNoResponse()` to match the pattern in `_commitUserTurn()`.

These can be committed together with the current fixes or as a follow-up in the same session.

## Strengths

1. **Root cause precision**: The fix introduces a distinct `acknowledgeNoResponse()` signal rather than inserting a fake ASSISTANT message, preserving the semantic contract of `onAssistantMessage()`.
2. **Minimal surface area**: One public method + one private helper. No new classes, providers, or abstractions (Principle #8).
3. **Guard discipline**: `acknowledgeNoResponse()` checks `_disposed` before phase, consistent with the orchestrator's defensive style in `stop()`, `onAssistantMessage()`, and `_updateState()`.
4. **ADR-0016 compliance**: `SessionNotifier` calls into `VoiceSessionOrchestrator` through the established provider-read pattern. Dependency direction preserved.
5. **Regression test quality**: Both tests would fail under the pre-fix code. The orchestrator test actively verifies the stuck-processing failure mode. The session test correctly asserts `isClosingComplete == true` (the right post-`endSession()` signal, not `activeSessionId == null` which is intentionally kept set for the closing UI).
6. **Regression ledger maintained**: Two new entries in `memory/bugs/regression-ledger.md` with complete root cause, fix, and test location.

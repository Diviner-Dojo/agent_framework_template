## Summary

Phase 2 implementation (Android assistant registration, onboarding flow, settings screen, navigation/intent routing) across 17 files with 133 tests at 86.6% coverage. Four specialists identified 3 blocking changes (all one-line defensive guards) and 7 advisory improvements. The implementation is architecturally sound with strong testability patterns, but missing defensive guards in session lifecycle methods create real risk of duplicate or orphaned sessions.

## Findings by Specialist

### QA Specialist
- Assistant auto-launch happy path (wasLaunchedAsAssistant=true + hasOnboarded=true) never exercised end-to-end in tests
- Greeting test asserts only `ListView` exists — vacuous assertion passes even with empty list
- `addPostFrameCallback` async chain may not drain before `pumpAndSettle` assertions
- Settings error branch, sendMessage null-session no-op, session list error branch all untested
- Confidence: 0.87

### Architecture Consultant
- `lastSessionDateProvider` duplicates `getAllSessionsByDate()` query — derive from `allSessionsProvider` stream instead
- `_checkAssistantLaunch` postFrameCallback lacks try/catch — startSession failure leaves app on /session with null state
- Missing ADRs for SharedPreferences choice and string-based routes decision
- `OnboardingNotifier.build()` uses `ref.watch` for non-reactive SharedPreferences — should be `ref.read`
- `onNewIntent` missing `setIntent(intent)` call (Android boilerplate)
- Confidence: 0.88

### Security Specialist
- Intent spoofing: exported activity accepts ACTION_ASSIST from any installed app, not just Android system
- `wasLaunchedAsAssistant` read-then-clear is not atomic — rapid second gesture silently dropped
- VOICE_ASSIST intent filter makes app appear in voice assistant chooser on pre-API29 without voice capability
- SharedPreferences onboarding flag unencrypted but appropriate (routing flag, not auth)
- Confidence: 0.88

### Independent Perspective
- `endSession()` has no concurrent-call guard — back-press during session ending creates duplicate messages
- `startSession()` has no active-session guard — rapid calls orphan the first session
- OEM settings fragmentation: ACTION_MANAGE_DEFAULT_APPS_SETTINGS unreliable on Samsung/Xiaomi/Huawei
- Platform channel timing race on cold start — wasLaunchedAsAssistant may be called before channel registered
- SharedPreferences may be cleared by aggressive OEM battery optimization
- Checkpoint system caught control-flow bugs but missed environmental failure modes
- Confidence: 0.82

## Required Changes Before Merge

1. **Add `endSession()` concurrent-call guard** [independent-perspective]
   In `lib/providers/session_providers.dart`, add at the top of `endSession()`:
   ```dart
   if (state.isSessionEnding) return;
   ```
   Prevents duplicate closing messages when back-press fires during an active `endSession()` call.

2. **Add `startSession()` active-session guard** [independent-perspective]
   In `lib/providers/session_providers.dart`, add at the top of `startSession()`:
   ```dart
   if (state.activeSessionId != null) return state.activeSessionId!;
   ```
   Prevents orphaned sessions from rapid assistant gestures or any code path that calls `startSession()` while a session is already active.

3. **Add try/catch in `_checkAssistantLaunch` postFrameCallback** [architecture-consultant]
   In `lib/app.dart`, wrap the callback body:
   ```dart
   WidgetsBinding.instance.addPostFrameCallback((_) async {
     try {
       await ref.read(sessionNotifierProvider.notifier).startSession();
       _navigatorKey.currentState?.pushNamed('/session');
     } catch (_) {
       // startSession failed — stay on initial route
     }
   });
   ```
   Prevents silent navigation to /session with null session state if startSession throws.

## Recommended Improvements (Non-Blocking)

1. **Intent spoofing mitigation** [security]: Add caller origin check in `MainActivity.kt` — verify `callingActivity?.packageName` is null or "android" before trusting ASSIST intents. Low practical risk for a personal journal app but raises the security floor.

2. **`onNewIntent` should call `setIntent(intent)`** [architecture]: Standard Android boilerplate. Add `setIntent(intent)` as first line of `onNewIntent` to keep `getIntent()` consistent.

3. **`ref.watch` → `ref.read` in `OnboardingNotifier.build()`** [architecture]: SharedPreferences is not reactive; `ref.watch` is semantically misleading. Change to `ref.read` for correctness.

4. **Strengthen greeting test assertion** [qa]: Replace `find.byType(ListView)` with a time-injected greeting text assertion to avoid vacuous pass.

5. **Add assistant auto-launch happy-path test** [qa]: Test that `wasLaunchedAsAssistant=true` + `hasOnboarded=true` actually pushes `/session` route.

6. **Add `endSession()` guard test** [qa]: After fixing blocking change #1, add a test that calls `endSession()` twice rapidly and verifies no duplicate messages.

7. **OEM-aware setup instructions** [independent]: Replace stock-Android instructions with "Open your phone's settings and search for 'Default assistant'" to improve setup success on Samsung/Xiaomi/Huawei devices.

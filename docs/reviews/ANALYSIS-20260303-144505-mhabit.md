## Recommended Adoptions

### NotificationService Abstract Interface (Score: 22/25)

- **What**: Abstract interface class declaring full notification contract. Factory dispatches to platform-specific implementations. FakeNotificationService enables test injection without mocking.
- **Where it goes**: `lib/services/notification_service.dart` (interface), `test/fakes/fake_notification_service.dart` (recording fake)
- **Why it scored high**: Fits our existing service-based architecture. Riverpod provider wrapping is natural. Enables testing of all notification-dependent code.
- **Implementation notes**: Discard singleton factory — use Riverpod provider. Place fake in test/ not lib/. Add call recording to fake (store scheduled/cancelled notifications for assertions). mhabit's const no-op fake is insufficient for meaningful tests.
- **Sightings**: 1 (first sighting)

### whenNeeded Data-Anchored Reminder Scheduling (Score: 23/25)

- **What**: Schedule notifications based on data state (last session date) rather than fixed recurrence. Fire only when user hasn't completed the target action. Complementary to Daily_You's entry-existence guard.
- **Where it goes**: Reminder scheduling logic in notification service
- **Why it scored high**: Directly resolves ADHD non-escalating requirement. Pre-mortem: without this, fixed-time reminders auto-disable after 3 dismissals, permanently losing the user.
- **Implementation notes**: Replace `lastUntrackDate` with "last session date" from SessionDao. Wire rescheduling trigger into session-logging flow. Replicate the 7 test cases from `habit_reminder_test.dart`. All scheduling functions should accept optional `DateTime? crtDate` parameter (default to AppClock.now()) for testability.
- **Sightings**: 1 (first sighting)

### Injectable AppClock (Score: 21/25)

- **What**: Singleton wrapping `DateTime.now()` with injectable replacement. Every scheduling function calls `AppClock().now()` instead of `DateTime.now()` directly.
- **Where it goes**: `lib/utils/app_clock.dart`
- **Why it scored high**: Prerequisite for testing any time-dependent logic. ~15 lines, no new dependency needed (skip quiver, use simple `DateTime Function()` field).
- **Implementation notes**: Implement without quiver dependency. Tests call `AppClock().setNow(() => fixedDateTime)`.
- **Sightings**: 1 (first sighting)

### Segmented Notification ID Namespace (Score: 19/25)

- **What**: Named constant ranges for notification IDs preventing silent collision between subsystems.
- **Where it goes**: `lib/services/notification_id_range.dart`
- **Why it scored high**: Prevents future collision bugs as notification channels grow.
- **Implementation notes**: Simplified version for our scope: one ID for daily reminder, reserved range for future AI-insight notifications. ~10 lines.
- **Sightings**: 1 (first sighting)

### HabitExporter Factory + Strategy + Mixin (Score: 21/25)

- **What**: Factory dispatches to ExportAll vs ExportFiltered. Shared serialization logic in mixin. Separate import class inverts the pattern.
- **Where it goes**: `lib/services/session_exporter.dart` (when data export is implemented)
- **Why it scored high**: Clean separation of scope selection from serialization. Maps directly to our "export all sessions vs. export by date range" requirement.
- **Implementation notes**: Replace sqflite `HabitDBCell` with Drift `Session` data class. Use `dao.getAllSessions()` / `dao.getSessionsByDateRange()`. The `archive` package handles ZIP for large exports.
- **Sightings**: 1 (first sighting)

## Anti-Patterns & Warnings

### Singleton Factory with Riverpod

- **What**: `static NotificationService? _instance` singleton persists across test runs
- **Where seen**: `notification_service.dart:103-118`
- **Why it's bad**: Conflicts with Riverpod's dependency graph, causes test pollution
- **Our safeguard**: Use Riverpod provider to manage NotificationService lifecycle

### FakeNotificationService Never Actually Tested

- **What**: The fake exists but is never used as a test double in mhabit's own test suite
- **Where seen**: `notification_service.dart:978-1053` (defined but unused in tests)
- **Why it's bad**: Architecture looks testable but the testing benefit is unvalidated
- **Our safeguard**: Write tests that actually use our fake from day one

## Deferred Patterns

### Per-Channel AppNotifyConfig (Score: 16/25)

- **What**: `Map<NotificationChannelId, bool>` enabling per-channel suppression at service layer
- **Why deferred**: Over-engineered for our current scope (1-2 notification channels). mhabit has 5+ channels justifying the pattern.
- **Revisit if**: We add a third notification channel (e.g., AI insights, weekly digest, sync status)

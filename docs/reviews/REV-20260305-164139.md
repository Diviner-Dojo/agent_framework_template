## Blocking Findings Resolved Before Review

### Pre-B1 — `_emptyRescheduleResult` Mutable Lists [QA + Arch — FIXED]

**Location**: `lib/services/notification_scheduler_service.dart:320–323`

**Description**: The `_emptyRescheduleResult` static final contained mutable `List` instances. Caller mutation would corrupt subsequent empty-guard returns (both QA and architecture-consultant flagged this independently). Fixed by wrapping with `List.unmodifiable()`.

**Fix applied**: Changed to `List<...>.unmodifiable([])` for both `rescheduled` and `failedTaskIds` fields.

## Advisory Findings

### QA

**A1** — Missing loop-skip test (all-null reminderTime with initialized service). Early-return path uses `_emptyRescheduleResult`; loop-skip path creates fresh lists. A guard change could accidentally return the shared sentinel from inside the loop. Add test: `'returns empty when all tasks have null reminderTime (loop-skip, not early-return)'`.

**A2** — `notificationBootRestoreProvider` `failedTaskIds` loop not tested at provider level. Service-level PlatformException catch is tested; the provider's `updateNotificationId(taskId, null)` call has zero unit coverage. Add `ProviderContainer` test with mock scheduler returning `failedTaskIds: ['t-fail-01']`.

**A3** — No multi-task partial failure test. Single-task test doesn't verify per-iteration catch isolation. Add two-task test: first succeeds, second throws PlatformException.

**A4** — `hasMissingDataWarning` test has redundant assertion. `insights.isNotEmpty` is implied by `any((i) => i.hasMissingDataWarning)`. Replace with `insights.where((i) => !i.hasMissingDataWarning).isNotEmpty` to specifically test narrative-distinct-from-warning behavior.

**A5 (low)** — `_ThrowingSchedulerService` passes `FlutterLocalNotificationsPlugin()` to super, registering the plugin singleton. Not a current issue since `scheduleNotification` is overridden; low flakiness risk only.

### Security

**A6 (low)** — `on PlatformException` over-catch accepted. Could clear a valid `notificationId` on a transient error (alarm loss, not data exfiltration). Already documented inline and in ADR-0033 §6. No action required.

**A7 (low)** — `@visibleForTesting` is static analysis, not runtime access control. Method compiled into release APK. Zero practical risk given no production callers and quality gate enforcement. Could add `avoid_calling_visiblefortesting_outside_test` lint rule to `analysis_options.yaml` for compile-time enforcement.

### UX

**A8 (medium)** — `textInputAction: TextInputAction.send` removes all keyboard newline paths. Hint text `'Type your thoughts…'` implies multi-thought entry, which contradicts submit-on-enter. Consider conditioning on `_isTextInputMode`: `send` in text-primary, `newline` in voice+text. Or update hint text to signal submit-on-enter behavior.

**A9 (medium)** — `'Values normalized to 0–1 (1 = highest recorded)'` subtitle uses technical vocabulary. `'Normalized'` is opaque; `'highest recorded'` may cause historical comparison (gap-adjacent). Suggested replacement: `'Chart scaled to your own range — top of chart is your personal highest'`.

**A10 (low)** — Correlation empty state: `'same dimensions'` is slightly technical. Improvement: `'once you have answered the same two questions on 5 or more separate days'`.

**A11 (low)** — Raw `r` value removed from correlation tiles — power users have no alternative. Consider long-press tooltip or expanded state in a future pass.

### Architecture

**A12** — ADR-0033 amendment should document why anonymous record was chosen over named class (e.g., `SyncResult` pattern). Key rationale: single consumer, no computed properties, structurally typed at call site. Add 2 sentences specifying that if `skippedTaskIds` or computed properties are added, the return type should be promoted to a named class.
